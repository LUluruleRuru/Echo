# Echo

> 一个生成口语化口播稿的 AI Skill。先讲，再写。幻灯片是口播稿的副产物，不是反过来。

## 这是什么

`Echo` 解决一个具体问题：AI 写的稿子念出来像机器人——太干净、太工整、太礼貌，3 秒就被听众识别为 AI 生成然后划走。

这个 Skill 控制的是 LLM 的行为，不是人类作者。它在写作过程中主动踩刹车——拦调研、打断框架搭建、阻止四版以上的反复修改——确保产出的是人说的话，不是文档念出来。

### SKILL.md

Skill 被调用时加载进 LLM 上下文。定义 LLM 必须做什么：

- **Core Pattern**：HOOK → GAP → BRIDGE → PROOF → ACTION 五段弧线，每段有硬时间约束和心理功能目标
- **Behavioral Rules**：写前校准、写后检查、危险信号中断、注意力紧急信号升级
- **Two Non-Negotiable Rules**：① 先讲再写 ② 找到张力再写任何东西

### scene-adaptation.md

数据层。SKILL.md 通过引用指向它。包含：

- **注意力预算表**：7 场景 × 5 节拍的秒数硬上限
- **心理功能层**：紧迫度等级质性信号、HOOK 三种模式（好奇心缺口 / 矛盾直给 / 身份锚定）、五节拍心理功能定义
- **失败条件矩阵**：按心理参与失败（不是信息迟到）重写的判据
- **场景适配**：七场景的节拍内容指南、张力类型、危险信号检测
- **诊断案例**：V4 失败案例 + 热门版女娲成功案例的逐句心理功能拆解

### 设计文档

两个 `.md` 文件记录了为什么这么设计——问题是什么、方案对比、决策边界。LLM 执行任务不需要读它们，但下次有人问"注意力预算为什么是 3 秒"或"为什么反问句不全禁止了"，答案在这里。

## 核心理念

### 三条原则

1. **口语先行，幻灯片后置**。先录音讲一遍，转文字，砍废话，最后反推幻灯片。这个顺序不能反。
2. **减法先行，加法后置**。每一版草稿写完必须先出减法版。砍 30-60%（按紧迫度）。
3. **决策先行，执行后置**。在不确定核心叙事的情况下做调研和搭框架，是浪费时间。

### 注意力预算系统

不同场景下，听众给主讲人的耐心完全不同。短视频 3 秒没打到痛点就划走，路演第 8 个项目投资人 10 秒不抬头就不会再抬头。每个场景 × 每个节拍有硬秒数上限——不是建议，超过上限意味着听众已经走了。

### 心理功能层（v3.0）

注意力预算的秒数约束是必要的，但不充分。判据不能只问"信息给到了没有"——要问"心理参与建起来了没有"。

**五节拍的心理功能**：

| Beat | 不是 | 而是 |
|------|------|------|
| HOOK | 引起注意 | **让听众倾身**——好奇（"然后呢？"）、认出自己（"这说的就是我"）、或被冲击（"什么？"） |
| GAP | 指出旧方法缺陷 | **让听众感到被看见**——"对，这就是我。而且这不是我的错。" |
| BRIDGE | 给出方案 | **给好奇心回报**——"那个矛盾，是这样解开的" |
| PROOF | 提供证据 | **让听众自己见证**——不是说服，是展示 |
| ACTION | 给下一步 | **让拥有感变得不可避免**——一个不可能失败的动作 + 占有欲 |

**HOOK 三种模式**：

| 模式 | 机制 | 防划走力 | 最佳场景 |
|------|------|:--:|------|
| 好奇心缺口 | 已知场景 + 反常识对比 → 听众在等答案 | 🟢 最强 | 短视频、路演 |
| 身份锚定 | 一个精准的具体行为 → "这说的就是我" | 🟢 强 | 短视频（垂直人群）、培训 |
| 矛盾直给 | 逻辑冲突 → "为什么会这样" | 🟡 中等 | 提案、答辩、汇报 |

### 反问句不是一律禁止

旧规则禁止所有反问句。v3.0 区分两种：

- **禁止**：代词假发现的问句（"你有没有发现…"）——听众没发现，失信
- **允许**：代言听众真实内心 OS 的问句（"怎么做到的？"）——听众确实在想这个，创造参与感

判别标准：**听众现在是不是确实在问这个问题？**

| 场景 | 紧迫度 | HOOK 上限 | 典型时长 | 示例 |

| 培训 | 🟢 低 | 30s | 28 分钟 | 内部培训 |

## 使用方式

### 在 Claude Code 中调用

```
/Echo
```

然后描述你的需求。Skill 会自动走校准流程——确认场景、时长、受众、证据水平、HOOK 心理策略——然后开始起草。

### 作为参考

如果你不是在 Claude Code 中使用，`SKILL.md` 和 `scene-adaptation.md` 可以作为任何 AI 写作工具的系统提示词参考。核心规则可以直接复制，场景表可以作为检查清单。

## 已知局限

1. **场景内状态变量未引入**。目前只按场景类型决定默认预算。同样是路演，当天第 1 个项目和第 8 个项目，听众耐心不同，但系统不区分——这属于设计上明确推迟的下一步。
2. **产品/项目引入时机未检查**。v3.0 的心理功能检查没有拦住"产品名字在 BRIDGE 才第一次出现"——V7 暴露了这个问题，V8 手动修复了它，但 Skill 的规则层尚未加入自动检查。
3. **中文特化**。4 字/秒的换算基于中文口播速度。英文约 2.5 词/秒。跨语言使用需要替换换算系数。
4. **未覆盖对话式场景**。播客对谈、访谈、圆桌讨论——这七个场景都是"一个人对一群人"，不适用于对话场景。
